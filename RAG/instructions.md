## Intelligent Customer Support
Intelligent Customer Support Knowledge Assistant (Weeks 1-5) – 40%

### Scenario:
You’re building a customer support assistant for “Omnia Retail Ltd” (fictional e-commerce company selling electronics). The system must handle real customer queries using a knowledge base of past tickets and company policies.

The dataset and instructions are included on the previous step.

### Deliverable 1A: Multi-turn Conversational RAG System (70% of Assignment 1)
Requirements:
1. Conversation Memory: Handle multi-turn conversations where context matters
    - Example: “I ordered a laptop” → “When will it arrive?” → “Can I change the address?”
    - System must remember order details across turns

2. Hybrid Retrieval Strategy: Combine multiple retrieval approaches
    - Semantic search for similar past issues
    - Keyword search for exact product names/order numbers
    - Metadata filtering (recency, category, satisfaction score)
    - Students must design and justify their hybrid approach

3. Dynamic Context Selection: Intelligently choose how much context to include
    - Not all retrieved documents are equally relevant
    - Must implement re-ranking or relevance scoring
    - Balance between context quality and token limits

4. Fallback Handling: Gracefully handle cases where RAG doesn’t help
    - Detect when no relevant information exists
    - Provide helpful fallback responses
    - Flag for human escalation with reasoning

#### The “Leap” from Labs:
- Labs cover single-turn RAG; assignment requires conversation memory
- Labs show basic retrieval; assignment requires hybrid strategy design
- Labs use simple top-k; assignment requires intelligent context selection
- Labs assume retrieval always works; assignment requires failure handling

### Deliverable 1B: Performance Optimisation & Business Metrics (30% of Assignment 1) – Choose One Only
Choose One of the Following:
Requirements:

1. Cost-Performance Optimisation: Minimise costs while maintaining quality
    - Compare local (Ollama) vs cloud (OpenAI) for different query types
    - Implement routing logic: simple queries → local, complex → cloud
    - Document cost-quality trade-offs with data

2. Business Metrics Dashboard: Track real support metrics
    - First response time (latency)
    - Resolution accuracy (using test set)
    - Customer satisfaction proxy (using LLM-as-judge)
    - Cost per interaction
    - Escalation rate (% queries needing human)

3. A/B Testing Report: Compare at least 2 system configurations
    - Different prompt strategies OR
    - Different retrieval parameters OR
    - Different embedding models
    - Present findings with statistical significance

4. Edge Case Documentation: Identify and document 10+ failure modes
    - What types of queries fail?
    - Why does the system struggle?
    - Proposed improvements

#### The “Leap” from Labs:
- Labs show basic evaluation; assignment requires business-focused metrics
- Labs use single model; assignment requires intelligent routing
- Labs evaluate once; assignment requires systematic A/B testing
- Labs work with clean examples; assignment requires edge case analysis
